# 订单-开票-回款闭环流程（重构版）

## 1. 目标与原则

- 目标：形成 `销售订单 -> 开票单据 -> 回款单据` 的可追溯闭环，支持“待开票/部分开票/已开票/待回款/部分回款/已结清”全流程管理。
- 原则：VBA 仅作为历史参考，不直接照搬实现；所有写入流程以服务端事务和一致性校验为准。
- 范围：仅覆盖财务管理中的开票单据、发票明细、回款单据页面与接口流程。

## 2. 数据对象与主关联

- 销售订单明细主键：`销售订单.订单ID`
- 开票表头主键：`开票单据.发票ID`
- 开票明细主键：`发票明细.明细ID`
- 回款明细主键：`回款单据.回款ID`

主关联链路：

1. `发票明细.发票ID -> 开票单据.发票ID`
2. `回款单据.明细ID -> 发票明细.明细ID`
3. `开票单据.客户ID -> 客户信息.客户ID`
4. `发票明细.项目编号 <-> 销售订单.项目编号`（业务关联）

## 3. 单据编号规则

- 开票单据：`KP-YYYYMMDD-XXX`
- 回款单据：`HK-YYYYMMDD-XXX`
- 序号按天递增、跨天重置。
- 编号由后端统一生成，前端只读，不允许手工改号。

## 4. 状态定义（统一口径）

### 4.1 开票状态（按订单明细）

- `待开票`：未产生发票明细。
- `部分开票`：发票明细累计金额 < 订单应开金额。
- `已开票`：发票明细累计金额 >= 订单应开金额。
- `已红冲`：对应发票明细/发票表头标记红冲。

### 4.2 回款状态（按发票明细）

- `待回款`：应收金额 > 0，且实收+贴息 = 0。
- `部分回款`：0 < 实收+贴息 < 发票金额。
- `已结清`：实收+贴息 >= 发票金额 或 `是否结清=1`。
- `异常`：孤儿回款、非法日期、金额逻辑冲突。

## 5. 页面流程（重构）

## 5.1 开票单据页面

1. 列表查询：
- 按单据编号、发票号码、客户、开票日期、状态筛选。
- 展开可见发票明细。

2. 新增开票：
- 从“可开票池”选择订单明细（默认仅显示未结清、可开票项）。
- 自动带出：项目编号、产品、合同号、客户信息、建议金额。
- 保存时服务端事务：
  - 写入开票表头；
  - 批量写入发票明细；
  - 回写状态。

3. 编辑开票：
- 可改：发票号码、开票日期、备注。
- 明细调整必须重算表头金额并校验一致性。

4. 红冲开票：
- 禁止物理删除。
- 通过“红冲”操作生成负向记录或标记红冲，并记录原发票ID。

## 5.2 回款单据页面

1. 列表查询：
- 按回款单号、客户、回款日期、回款方式、状态筛选。

2. 新增回款：
- 从“未结清发票明细池”选择明细。
- 录入：实收金额、贴息金额、回款日期、回款方式、收款账户。
- 保存时服务端事务：
  - 写入回款明细（同单据编号可多行）；
  - 按 `明细ID` 计算剩余应收并更新结清状态（回款+发票明细）。

3. 编辑回款：
- 允许改单据未审核明细，修改后重新结清计算。

4. 作废/冲销：
- 禁止物理删除。
- 以状态或冲销记录处理，并保留审计日志。

## 6. 关键校验规则（必须后端执行）

1. 金额一致性：
- 开票保存时必须校验：`开票单据.金额 = SUM(发票明细.金额)`（误差阈值 0.01）。

2. 回款金额合法性：
- 单行 `实收金额 >= 0`，`贴息金额 >= 0`。
- 累计 `(实收+贴息)` 不得超出发票明细金额（是否允许超收由配置决定）。

3. 日期合法性：
- 单据日期、开票日期、回款日期必须为合法业务日期，禁止小于 `2000-01-01`。

4. 引用完整性：
- 回款写入前必须校验 `明细ID` 在发票明细存在。
- 有回款的发票明细禁止删除（仅允许红冲/作废流程）。

5. 客户一致性：
- 同一单据下明细客户必须一致。

## 7. 异常治理策略（针对已发现问题）

1. 孤儿回款（回款有、发票明细无）：
- 标记为 `异常`，进入异常台账。
- 提供“补挂发票明细ID”或“冲销回款”工具，不允许继续参与结清计算。

2. 异常日期（如 `0103-12-14`）：
- 入库前拦截；
- 存量数据落异常清单，需人工修正后恢复业务状态。

3. 表头金额与明细汇总不一致：
- 接口层阻断保存；
- 存量差异进入“金额差异台账”，逐条修复。

## 8. 实施顺序（建议）

1. 第一步（只读上线）：
- 开票/回款页面改为真实接口查询，替换前端 mock。

2. 第二步（写入上线）：
- 开票新增/编辑、回款新增/编辑全部改为事务写入。

3. 第三步（闭环完善）：
- 状态机、异常台账、红冲/冲销、审计日志。

4. 第四步（治理收口）：
- 清理孤儿回款、异常日期、金额差异历史数据。

## 9. 与 VBA 的差异约束（明确）

- 不允许“覆盖保存=先删后插”直接落库。
- 不允许删除已参与回款链路的数据。
- 不使用 `IDENT_CURRENT` 取新ID；统一使用安全返回主键方式。
- 所有关键写入必须事务化，失败全量回滚。
